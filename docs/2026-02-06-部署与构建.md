# NanoMind 部署与构建文档

> 描述项目的构建流程、打包策略、部署方式和开发环境配置。

---

## 一、架构概览

### 1.1 整体架构

```
┌─────────────────────────────────────────────┐
│              单个可执行文件 (nanomind)          │
│  ┌───────────────┐  ┌────────────────────┐  │
│  │   后端服务      │  │  嵌入的前端静态资源   │  │
│  │  (HTTP API)    │  │  (HTML/CSS/JS)     │  │
│  └───────┬───────┘  └────────┬───────────┘  │
│          │                    │               │
│          └──────┬─────────────┘               │
│                 │                              │
│          ┌──────┴──────┐                      │
│          │   HTTP 服务   │                      │
│          │  /api/* → API │                      │
│          │  /* → 前端     │                      │
│          └──────────────┘                      │
└─────────────────────────────────────────────┘
                    │
                    ▼
        ┌──────────────────┐
        │  本地文件系统       │
        │  ~/.nanomind/memos │
        └──────────────────┘
```

### 1.2 核心设计思路

- **单二进制**：前端构建产物在编译时嵌入到后端二进制中
- **零外部依赖**：不依赖数据库、Redis、Docker 等外部服务
- **即插即用**：下载二进制文件，直接运行即可
- **前后端分离开发**：开发时前后端独立运行，生产时合为一体

## 二、前端嵌入策略

### 2.1 嵌入机制

后端使用语言自带的静态资源嵌入能力，将前端构建产物（`dist/` 目录）嵌入到编译后的二进制文件中。

**关键点**：
- 嵌入时包含隐藏文件（如 `.vite` manifest 文件）
- 提供文件系统接口，去除 `dist/` 前缀（即 `dist/index.html` 变为 `index.html`）

### 2.2 静态文件服务

嵌入的前端文件通过 HTTP 服务提供：

1. **路径匹配**：请求路径映射到嵌入的文件系统
2. **根路径**：`/` 映射到 `index.html`
3. **Content-Type**：根据文件扩展名自动设置
4. **SPA 回退**：非 API、非静态文件的请求返回 `index.html`
5. **扩展名判断**：
   - 无扩展名的路径 → 返回 `index.html`（支持前端路由）
   - 有扩展名但文件不存在 → 返回 404

## 三、构建流程

### 3.1 前端构建

```
安装依赖 → 构建产物 → 输出到 dist/ 目录
```

**构建配置要点**：
- 路径别名：`@` → `./src`（简化导入路径）
- 跳过类型检查（加速构建）
- 输出目录：`site/dist/`

### 3.2 后端构建

```
编译后端代码 → 嵌入 dist/ 静态资源 → 输出单个二进制
```

**构建配置要点**：
- 禁用 CGO（`CGO_ENABLED=0`），确保静态链接
- 链接标志：`-s -w`（去除调试信息和符号表，减小体积）
- 输出目录：`bin/`

### 3.3 完整构建

```
前端构建 → 后端构建（自动嵌入前端产物）
```

**注意**：后端构建必须在前端构建之后执行，因为嵌入指令指向 `dist/` 目录。

## 四、跨平台构建

### 4.1 支持的目标平台

| 平台 | 架构 | 输出文件名 |
|------|------|-----------|
| Linux | amd64 | `nanomind-linux-amd64` |
| Linux | arm64 | `nanomind-linux-arm64` |
| macOS | amd64 (Intel) | `nanomind-darwin-amd64` |
| macOS | arm64 (Apple Silicon) | `nanomind-darwin-arm64` |
| Windows | amd64 | `nanomind-windows-amd64.exe` |

### 4.2 交叉编译

所有平台均可通过设置目标操作系统和架构环境变量实现交叉编译：
- 不依赖 CGO，纯静态编译
- 在任何平台上都可以构建所有目标平台的二进制

## 五、开发环境

### 5.1 开发模式架构

开发时前后端独立运行：

```
浏览器 ──→ Vite Dev Server (5173)
              │
              ├─ 前端热更新
              └─ /api/* ──代理──→ Go Backend (8080)
                                     │
                                     └─ 文件系统操作
```

### 5.2 前端开发服务

- 端口：5173
- 热模块替换（HMR）
- API 代理：`/api` → `http://localhost:8080`
- 自动打开浏览器（可选）

### 5.3 后端开发服务

- 端口：8080
- 直接运行源码（无需编译）
- CORS 中间件（允许跨域，开发时前端在不同端口）
- 请求日志
- Panic 恢复

### 5.4 联合开发

同时启动前后端：
1. 先启动后端（端口 8080）
2. 等待 1 秒后启动前端（端口 5173）
3. 通过 trap 信号确保两个进程同时退出

### 5.5 快捷命令清单

| 命令 | 功能 |
|------|------|
| `install` | 安装前端依赖 |
| `dev` | 同时启动前后端（推荐） |
| `dev-web` | 仅启动前端 |
| `dev-backend` | 仅启动后端 |
| `build` | 构建前端 + 后端 |
| `build-web` | 仅构建前端 |
| `build-backend` | 仅构建后端（当前平台） |
| `build-all` | 构建所有平台 |
| `clean` | 清理构建产物 |
| `run` | 运行已编译的二进制 |
| `test` | 运行所有测试 |

## 六、运行时配置

### 6.1 环境变量

| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `MEMOS_PATH` | 笔记文件存储目录 | `~/.nanomind/memos` |
| `PORT` | HTTP 服务端口 | `8080` |

### 6.2 默认行为

- 如果 `MEMOS_PATH` 目录不存在，首次写入时自动创建
- 如果无法获取用户 Home 目录，回退到 `./memos`（当前目录下）
- 启动时在日志中打印实际使用的笔记目录路径

## 七、部署方案

### 7.1 最简部署（推荐）

```bash
# 下载对应平台的二进制文件
# 直接运行
./nanomind

# 自定义配置
MEMOS_PATH=/data/notes PORT=3000 ./nanomind
```

### 7.2 Systemd 服务（Linux 推荐）

创建 systemd 服务单元文件，实现开机自启、自动重启：

关键配置项：
- 工作目录
- 环境变量（MEMOS_PATH、PORT）
- 自动重启策略
- 日志输出

### 7.3 Docker 部署（可选扩展）

当前项目未包含 Dockerfile，但部署非常简单：
- 基础镜像：scratch 或 alpine
- 仅需复制单个二进制文件
- 暴露端口
- 挂载数据卷（笔记目录）

### 7.4 反向代理

可以配合 Nginx/Caddy 等反向代理：
- 提供 HTTPS
- 添加基础认证（HTTP Basic Auth）
- 域名绑定
- 静态资源缓存

## 八、项目目录结构建议

```
nanomind/
├── cmd/                    # 应用入口
│   └── app/
│       └── main.go        # 主程序入口
├── internal/               # 内部包（不对外暴露）
│   └── api/
│       └── handlers.go    # API 处理器
├── pkg/                    # 公共包
│   └── siteserver/
│       └── siteserver.go  # 静态文件服务
├── site/                   # 前端项目
│   ├── embed.go           # 嵌入指令
│   ├── src/               # 前端源码
│   ├── dist/              # 前端构建产物（git ignore）
│   └── package.json
├── bin/                    # 编译输出（git ignore）
├── docs/                   # 文档
├── Makefile                # 构建脚本
├── go.mod
└── go.sum
```

**设计原则**：
- `cmd/`：应用入口，尽量薄，只做初始化和组装
- `internal/`：业务逻辑，不可被外部包引用
- `pkg/`：可复用的工具库
- `site/`：前端完全独立，通过 `embed.go` 桥接

## 十、关键技术决策记录

以下是项目中做出的关键技术决策，供重构时参考：

### 10.1 为什么不用数据库？

- 目标是极简部署，不依赖外部服务
- Markdown 文件本身就是数据，无需转换存储
- 用户可以直接用文件管理器或其他编辑器操作笔记
- 文件系统天然支持目录结构

### 10.2 为什么采用 Markdown 存储 + 富文本编辑？

- Markdown 是最通用的笔记格式，可移植性强
- 但纯 Markdown 编辑体验不如所见即所得
- 折中方案：编辑时使用富文本编辑器，存储时转为 Markdown
- 代价：部分富文本格式（字体大小、颜色等）无法完美映射到标准 Markdown

### 10.3 为什么采用单二进制部署？

- 降低部署门槛，适合非技术用户
- 无需安装运行时环境
- 方便在各种环境（NAS、VPS、树莓派）上运行
- 更新简单：替换文件即可

### 10.4 为什么不做用户认证？

- 定位为自托管应用，通常在内网或通过 VPN 访问
- 认证功能可以通过反向代理（Nginx Basic Auth）实现
- 保持核心功能简单，避免引入用户管理复杂性
